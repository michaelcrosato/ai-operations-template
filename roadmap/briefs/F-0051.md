## Design contract (follow exactly; deviations = escalate)
Verb (house style — positional args, trailing free-text reason, mirroring `--status <id> blocked <reason...>`):

    npx ts-node scripts/update-state.ts --amend <id> <field> '<json-value>' <reason...>

Guards, in this order, each with a distinct `fail()` message:
1. Field allowlist: `const AMENDABLE = ['title', 'description', 'acceptance'];` — closed constant with a KNOWN_KEYS-style comment ("extend deliberately"). Any other field → fail (this is what keeps `--amend` away from the evidence-gated flip contract: id/status/passes/evidence/attempts/paths/priority/tier/dependencies/spec_ref all rejected).
2. Status allowlist: the row's status must be `in_progress` or `done`. pending/blocked rows are re-groomed via the existing `--remove` + `--add` path — fail with a message saying exactly that.
3. Value shape (checked before mutation, for a clean error): title/description → non-empty JSON string; acceptance → non-empty JSON array of non-empty strings. The value argument is parsed with JSON.parse (fail on parse error, like `--add`).
4. Reason: join the trailing args with spaces; must be non-empty AND match `/DECISIONS(\.md)?\s+\d{4}-\d{2}-\d{2}/` — every amendment must cite a dated DECISIONS entry. Fail message names the required pattern.
5. Logging: on success print `[update-state] AMEND <id>.<field>` plus the old value and the new value (JSON.stringify both) and the reason.
6. Persistence: mutate the row, then `save(data)` (which re-validates the whole backlog — that plus guard 3 satisfies AC3). No schema change, no new row field, no KNOWN_KEYS change.
